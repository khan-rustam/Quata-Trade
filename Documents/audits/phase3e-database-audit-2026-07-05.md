# Phase 3E — Database Audit (PostgreSQL 16 / Kysely)

**Date:** 2026-07-05 · **Method:** 6 parallel senior-DBRE readers over migrations, schema/FKs, constraints/triggers, indexes/performance, concurrency/locks, and ledger/audit-chain integrity → adversarial verification of critical/high data-integrity findings → hard-calibrated scoring against a production custodial-ledger bar. All findings cite real migration SQL / query code.
**Verdict:** **QUALIFIED PASS — integrity under concurrency is guaranteed at pilot scale by *verified construction*, not assertion. It is NOT yet guaranteed at scale or under the multi-asset roadmap** without a bounded, non-architectural remediation set. 3E clears the *integrity* gate; it does not clear the *scale* gate.

## What was verified SOUND (the concurrency core is the system's strongest part)

- `postJournal` locks `account_balances` `FOR UPDATE` in **globally-sorted order** (`ledger.service.ts:141-147`) → deadlock-free; cross-resource order (trade→offer→balance) is consistent everywhere; **oversell is impossible**.
- **Deposit credit is exactly-once** (row lock + status guard + UNIQUE journal idempotency key + savepoint recovery atop ON CONFLICT). Confirm-vs-expiry collapses to exactly one terminal state. Withdrawal daily-cap TOCTOU is closed with a **per-user advisory xact lock**.
- **No signer, RPC, argon2, or MinIO call ever runs inside a money transaction** — critical for a custodian. No data-integrity race was found.
- The READ COMMITTED + pessimistic-lock choice (Deviation D14, vs the docs' SERIALIZABLE) is **sound for this access pattern** (docs should be reconciled to match).
- **Append-only is enforced twice** (RULEs + role REVOKE); the double-entry trigger fires on every INSERT path; the append surfaces (ledger, `trades.status`, `audit_logs`) are grep-verified to be written by exactly one service each.
- **The reconciliation kill-switch is real and self-healing:** every 10 min it recomputes `SUM(entries)` vs cached balances and, on any mismatch, flips `withdrawals_paused=true`, honored on both the entry path and the broadcaster.

## Integrity backstop holes (dormant — none reachable by shipped code, none moves money, but a ledger is graded on its backstops)

**E-1 · The deferred balanced-journal trigger sums `amount` ignoring per-row `asset`** (`0002_ledger.ts:57-68`). Harmless on single-asset USDT, but at asset #2 a `+1 BTC / −1 USDT` journal nets to a **false zero** and the primary double-entry backstop passes an unbalanced entry. **The single most important thing to fix before multi-asset:** rewrite to assert zero-sum PER asset (`GROUP BY asset HAVING SUM(amount)<>0`) and add a composite FK/guard that `ledger_entries.asset` equals the account's asset.

**E-2 · `trg_trade_fsm` guards only `BEFORE UPDATE OF status`** (`0004_trading.ts:99`) — a row can be **INSERTed directly into a terminal status** (COMPLETED/RESOLVED_*), bypassing the FSM backstop. No shipped path reaches it and reconciliation flags the null `release_journal_id`, but any consumer trusting `trades.status` without re-deriving from `trade_events` sees corrupt state. **Fix:** BEFORE INSERT trigger forcing `status='OPENED'` (+ partial constraints tying `escrow_journal_id`/`release_journal_id` to status).

**E-3 · `big_needs_two` dual-approval CHECK hardcodes `500000000`** (`0003_wallet.ts:69`) while the operative `dual_approval_threshold` is admin-configurable — the last-line dual-control guard and enforced policy **diverge in both directions** (mirrors 3B-M3). **Fix:** bound the setting ≤ the CHECK constant (or move enforcement to a trigger reading settings); also make the clause NULL-safe (`approved_by IS NOT NULL AND second_approver IS NOT NULL AND second_approver <> approved_by`).

**E-4 · Audit `seq` is `GENERATED BY DEFAULT` (not `ALWAYS`) and the hash chain is keyless** (`0008_audit_chain_seq.ts:21`). Overridable identity risks a UNIQUE-collision DoS on the in-transaction audit write, and a keyless chain cannot detect forged appends by anyone with write access. **Fix:** `GENERATED ALWAYS AS IDENTITY`; key the chain with an HMAC secret held outside the DB.

## Performance / scale (the weak axis — availability, not corruption)

**E-P1 (sharpest edge) · `verifyChain()` loads the *entire* unbounded `audit_logs` table into the Node heap on a synchronous admin request** (`audit.service.ts:101-127`), then SHA-256s every row in a blocking loop. At 5–10M rows this **OOM-kills the whole API process** (not just the endpoint), blocks the event loop, and pins one of 20 pooled connections with no `statement_timeout` — failing exactly when the log is largest and proof-of-non-tampering matters most. **Fix:** checkpoint a "last verified seq + head hash" and verify only the delta via keyset-paginated streaming in a worker job.

- **No partitioning/archival on 7 unbounded high-write tables** (`ledger_entries`, `trade_events`, `trade_messages`, `audit_logs`, `risk_events`, `notifications`, `outbox`) → monotonic degradation; several are append-only and un-prunable by the app. **Fix:** declarative range-partition by `created_at`/month.
- **Worker hot-loops full-scan growing tables:** email-send scans `notifications` every 30s with **no matching index** (add partial `(created_at) WHERE channel='email' AND status='queued'`); password-reset scans `auth_tokens.token_hash` **unindexed**; `kyc_submissions.user_id` is an **unindexed FK**; the deposit scanner fans **one RPC per active address** every 30s (switch to block-range/batched scanning). Trade-room chat loads **all** messages unbounded with an N+1 to MinIO. Reconciliation does a **full-table GROUP BY every 10 min** (reconcile incrementally on changed `version`).
- **No `statement_timeout` / `lock_timeout` / `idle_in_transaction_session_timeout`** on the 20-connection pool — the cheap, high-leverage backstop that would downgrade E-P1, the advisory-lock stall, and pool-starvation from incidents to bounded errors.

## Concurrency liveness deductions (no corruption, but throughput/availability risk)

- **The global audit-chain advisory lock is held to COMMIT inside every audited money transaction** (`audit.service.ts:55`) while hot `platform`/`external` balance rows are `FOR UPDATE`-locked → a single stalled backend (GC/CPU) freezes the whole audited money path. **Fix:** move audit append to a short separate committed tx (or a monotonic-sequence scheme), not inside the money tx.
- **`ScreeningService.check()` runs on the pool connection (not the tx) while the deposit-credit row lock is held** (`deposit-confirmation.service.ts:116`) → pool-starvation/self-deadlock under load. **Fix:** thread the transaction into `check()`.
- Hot singleton `platform_treasury`/`external` balance rows serialize + bloat at very high throughput (stripe/derive-at-read + aggressive autovacuum).

## Migrations correctness

- **Duplicate `0011_` prefix** (`0011_content`, `0011_withdrawal_addresses`) — deterministic *today* only by alphabetical luck (JS `files.sort()` + Kysely's name sort both place `content` first); a future `0011_*` that sorts earlier would trip Kysely's ordered-migration corruption check and **block deploys on every already-migrated environment**. Renaming now is only safe *before* it is recorded in `kysely_migration` anywhere — after that the recorded name is frozen. Add a CI check rejecting duplicate prefixes.
- **`.catch(() => {})` on GRANT/REVOKE is ineffective** — Kysely runs the whole pending batch in ONE transaction, so a caught SQL error still poisons the transaction and the *next* statement fails with an opaque "current transaction is aborted." The resilience these wrappers promise doesn't exist (dead code giving false confidence — a "silent assumption" the project's own rule #1 forbids). **Fix:** use `IF NOT EXISTS`/server-side `DO $$ … IF EXISTS`, derive DB name via `current_database()`.
- **`0007` down() is broken** — `DROP ROLE quatatrade_readonly` fails because the `ALTER DEFAULT PRIVILEGES` granted in up() is never revoked (pg_shdepend blocks the drop). App-role privileges are point-in-time with no `ALTER DEFAULT PRIVILEGES` safety net, and **roles created in-migration do not survive a database-level restore**. Role passwords are interpolated into raw SQL with single-quote-only escaping and default to weak dev secrets — fail fast if unset in non-dev.

## Schema

- **Fiat columns hardcode `price_xaf_per_unit`/`fiat_amount_xaf` but now hold arbitrary per-country currency** (post multi-country) with **no per-row currency code** — stamp an immutable `currency_code` (+ `fiat_decimals`) snapshot at creation and drop the `_xaf` naming.
- `dispute_evidence.submitted_by` is a bare uuid with **no FK** (orphan-row risk on dispute provenance); the **same-market invariant has only an app-layer guard, no DB backstop**; `notifications.status` is unconstrained text; minor unindexed non-money FKs (`settings.updated_by`, `sessions.rotated_from`).

## Remediation gating the scale phase (none is a redesign)

1. Per-asset balanced-journal trigger (E-1) — **before any second asset**.
2. Bound `verifyChain`/reconciliation with streaming/windowing (E-P1).
3. Add pool `statement_timeout` / `lock_timeout` / `idle_in_transaction_session_timeout`.
4. Move screening off the locked path; move the audit-chain lock out of the money tx.
5. Reconcile `big_needs_two` with the settings threshold; tighten the FSM-INSERT and audit-`seq` backstops.
6. Add the handful of worker-path indexes + a retention/partitioning plan.
7. Resolve the duplicate `0011_` prefix (and fix the broken `0007` down / ineffective GRANT catches).

The foundation is sound; the concurrency/integrity core is well-engineered and holds by construction. Ship-to-pilot: yes. Before scale or a second asset: the above is mandatory.
